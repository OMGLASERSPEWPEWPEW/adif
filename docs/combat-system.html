The Kill-and-Loot Packet Timeline

C→S OP_TargetMouse (4 bytes: entity_id) — Player clicks NPC
C→S OP_Consider (28 bytes: target_id) — Player right-clicks /con
S→C OP_Consider (28 bytes: faction, con color, HP) — Server response
C→S OP_AutoAttack (4 bytes: 0x01) — Player presses attack
S→C OP_Damage (23 bytes Titanium) — Each melee hit/miss
S→C OP_MobHealth (3 bytes: HP %) — Broadcast to nearby
S→C OP_HPUpdate (10 bytes) — Player's own HP bar
S→C OP_Death (32 bytes) — NPC dies, corpse created
C→S OP_LootRequest (4 bytes: corpse_id) — Player opens corpse
S→C OP_MoneyOnCorpse (20 bytes) — Opens loot window + coins
S→C OP_ItemPacket (variable) — One per loot item
S→C OP_LootRequest (4 bytes: echo) — Client requires echo
C→S OP_LootItem (12 bytes) — Player takes item
S→C OP_LootItem (12 bytes: echo ACK) — Confirm pickup
C→S OP_EndLootRequest (4 bytes) — Close loot window
S→C OP_LootComplete (0 bytes) — Done

Targeting

OP_TargetMouse (0x6C47) — Mouse Click Target

Client sends 4-byte ClientTarget_Struct with the entity ID of the clicked mob.

Server behavior: Sets internal target, un-targets previous. No response packet is sent. The client already knows its target.

Bridge implementation: Store cs.target_id from the packet. No response needed.

Consider

OP_Consider (0x65CA) — Right-click /con

Request (C→S, 28 bytes)

struct Consider_Struct {
  u32 playerid;    // 0: (unused in request)
  u32 targetid;    // 4: entity ID to consider
  u32 faction;     // 8: (unused in request)
  u32 level;       // 12: (unused in request)
  i32 cur_hp;      // 16: (unused in request)
  i32 max_hp;      // 20: (unused in request)
  u8  pvpcon;      // 24: (unused in request)
  u8  pad[3];      // 25: padding
}; // 28 bytes

Response (S→C, 28 bytes)

struct Consider_Struct {
  u32 playerid;    // player's entity ID
  u32 targetid;    // target's entity ID
  u32 faction;     // faction standing (see below)
  u32 level;       // con color (see below)
  i32 cur_hp;      // target current HP
  i32 max_hp;      // target max HP
  u8  pvpcon;      // 0 for PvE
  u8  pad[3];
};

Melee Combat Flow

OP_AutoAttack (0x5E55) — Toggle Auto-Attack

Client sends uint32: 1 = start attacking, 0 = stop. No server response.

Server starts melee round timer based on weapon delay (primary hand). Each timer tick = one attack round.

OP_Damage (0x5C78) — Damage Event

// Titanium CombatDamage_Struct -- 23 bytes (NO special field)
struct CombatDamage_Struct {
  u16 target;       // 0: target entity ID
  u16 source;       // 2: attacker entity ID
  u8  type;         // 4: damage type (0=1HS, 1=1HB, 2=2HS, 36=pierce, 28=H2H, 0xE7=spell)
  u16 spellid;      // 5: spell ID (0 for melee)
  u32 damage;       // 7: damage amount (0 = miss)
  f32 force;        // 11: melee push force
  f32 hit_heading;  // 15: attacker heading
  f32 hit_pitch;    // 19: pitch
}; // 23 bytes total

Broadcast to all nearby clients via QueueCloseClients() after each hit.

HP Updates After Damage

OP_HPUpdate (0x3BCF) — To Self

u32 cur_hp;     // exact HP
i32 max_hp;
i16 spawn_id;   // 10 bytes

OP_MobHealth (0x0695) — To Others

i16 spawn_id;
u8  hp;         // HP percentage 0-100
              // 3 bytes

Melee Round Sequence (EQEmu)

  1. Check auto_attack == true && attack_timer.Check()
  2. Validate: target alive, in LOS, within combat range
  3. DoAttackRounds(target, slotPrimary):
    • Primary hand attack (hit/miss roll, damage calculation)
    • Check double attack skill → extra swing
    • Check triple attack → extra swing
    • Check flurry → extra swing
  4. If dual wield && attack_dw_timer.Check(): DoAttackRounds(target, slotSecondary)
  5. Each hit: send OP_Damage, update HP, broadcast OP_MobHealth

NPC Death & Corpse

OP_Death (0x6160) — Death Broadcast

struct Death_Struct {
  u32 spawn_id;      // 0: who died
  u32 killer_id;     // 4: who killed them
  u32 corpseid;      // 8: corpse entity ID (= spawn_id for NPCs)
  u32 bindzoneid;    // 12: bind zone (0 for NPCs)
  u32 spell_id;      // 16: killing spell (0xFFFFFFFF = melee)
  u32 attack_skill;  // 20: damage type
  u32 damage;        // 24: killing blow damage
  u32 unknown028;    // 28: padding
}; // 32 bytes

Key insight: The corpse inherits the NPC's entity ID. No OP_DeleteSpawn + OP_NewSpawn needed. The client renders the death animation and switches to corpse state automatically from OP_Death.

Death Sequence (EQEmu)

  1. NPC HP reaches 0 via CommonDamage()
  2. Broadcast OP_Death to all clients
  3. Compute XP distribution (solo/group)
  4. Create Corpse object from NPC's loot table (inherits entity ID)
  5. Add loot items from loottable → loottable_entries → lootdrop → lootdrop_entries → items
  6. Add coins (mincash/maxcash from loottable)
  7. Remove NPC entity, add Corpse entity (same ID)
  8. Start corpse decay timer

Loot Data Chain (PostgreSQL)

npc_types.loottable_id (e.g., 102975 for Grobb Bashers)
    → loottable (26,514 rows) — mincash, maxcash, avgcoin
        → loottable_entries (55,339 rows) — lootdrop_id, probability, multiplier
            → lootdrop (51,943 rows) — name
                → lootdrop_entries (482,577 rows) — item_id, chance
                    → items (117,944 rows) — full item data

Example: Basher_Banzuk (loottable 102975):
  • 50% Intestine Necklace (item 13311)
  • 66.7% Round Shield (item 9002)
  • 100% Long Sword (item 5002)

Loot Window Flow

Step 1: OP_LootRequest (0x6F90) — Open Corpse

Client sends uint32 corpse entity ID. Server validates distance (≤25 units), permissions, not already being looted.

Step 2: OP_MoneyOnCorpse (0x7FE4) — Opens Loot Window

struct moneyOnCorpseStruct {
  u8  response;    // 0: LootResponse enum (1=Normal, 5=TooFar)
  u8  unknown1;    // 1: 0x42 on success, 0x5a on error
  u8  unknown2;    // 2: 0xef on success, 0x40 on error
  u8  unknown3;    // 3: 0
  u32 platinum;    // 4-7
  u32 gold;        // 8-11
  u32 silver;      // 12-15
  u32 copper;      // 16-19
}; // 20 bytes

This packet is what opens the loot window on the client. Even if there's no money, it must be sent with response=1.

Step 3: OP_ItemPacket (0x3397) — Send Each Item

One packet per loot slot. Uses the same Titanium pipe-delimited text serialization as CharInventory (we already have serialize_titanium_item()).

The loot slot ID is used instead of the inventory slot ID in the item's slot field.

Step 4: Echo OP_LootRequest Back

EQEmu source comment: "Disgrace: Client seems to require that we send the packet back." The client hangs without this echo.

Step 5: OP_LootItem (0x7081) — Pick Up Item

struct LootingItem_Struct {
  u32 lootee;      // corpse entity ID
  u32 looter;      // player entity ID
  u16 slot_id;     // corpse loot slot
  u8  pad[2];
  i32 auto_loot;   // >0 for auto-loot
}; // 12 bytes

Server echoes this packet back as ACK, moves item to player inventory.

Step 6: OP_EndLootRequest / OP_LootComplete

Client sends EndLootRequest (uint32 corpse_id). Server sends LootComplete (0 bytes). If corpse is now empty, server sends OP_DeleteSpawn (4 bytes).

Implementation Plan — 4 Phases

Phase 1: Targeting & Consider (1 session)

Simplest combat feature. One opcode in, one response out. Proves the targeting pipeline works.

  1. Store target_id in ClientState when OP_TargetMouse received
  2. Keep a HashMap of spawned NPC data (entity_id → level, race, class, hp) during zone entry
  3. On OP_Consider: look up NPC by target_id, compute con color from level difference, build 28-byte response
  4. Default faction to INDIFFERENTLY (5) — full faction system deferred

Phase 2: Auto-Attack & Damage (2-3 sessions)

The core combat loop. Player can attack and see damage numbers.

  1. Add OP_AutoAttack handler — store auto_attack flag + target in ClientState
  2. Implement server-side melee timer (weapon delay from equipped weapon or default 2000ms)
  3. On each melee round: hit/miss roll, damage calculation (simple: player_level * 2 vs NPC AC)
  4. Send OP_Damage (23 bytes) to client for each hit
  5. Send OP_MobHealth (3 bytes) after HP changes
  6. Track NPC HP in bridge (reduce on each hit)
  7. NPC can also hit player — send OP_Damage from NPC source to player target
  8. Send OP_HPUpdate (10 bytes) when player takes damage

Phase 3: Death & Corpse — DONE (2026-06-30)

NPC death with corpse transition and XP award.

  1. OP_Death (32 bytes) sent on NPC HP ≤ 0
  2. Corpse inherits entity ID — is_corpse flag on SpawnedNpcInfo
  3. OP_ExpUpdate sent with XP gain (level * 15)
  4. Auto-attack stops, combat tick skips corpses

Phase 4: Looting — DONE (2026-06-30)

Full loot window with DB-backed loot resolution. Reuses serialize_titanium_item().

  1. OP_LootRequest: validate corpse, lazy-resolve loot from 5-table DB chain
  2. OP_MoneyOnCorpse: opens loot window with coins (mincash/maxcash from loottable)
  3. OP_ItemPacket for each item (probability/chance rolls per lootdrop group)
  4. Echo OP_LootRequest back (client requires it)
  5. OP_LootItem: remove from corpse, echo ACK
  6. OP_EndLootRequest: LootComplete + OP_DeleteSpawn for empty corpses

Decision: Bridge-Only vs Full Zone Server

Two approaches for implementing combat:

Option A: Bridge-Only (Recommended for now)

Implement combat directly in the bridge (main.rs). The bridge already tracks NPCs, has DB access, and manages the UDP session. Add melee timer, HP tracking, and loot state to the bridge. Faster to ship, single process.

Option B: Full ECS Integration (Future)

Wire the bridge to the adif-zone ECS server via protobuf. Bridge translates Titanium opcodes → protobuf. Zone server runs combat in ECS and sends results back. Proper architecture but requires the full TCP pipeline between bridge and zone.
