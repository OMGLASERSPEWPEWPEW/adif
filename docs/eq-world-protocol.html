Packet Capture Analysis

EQ Titanium Client-Server Protocol

Complete protocol documentation from packet capture data. Login, world, and zone phases dissected byte-by-byte. Used to build the ADIF bridge server.

3 server ports UDP transport DES CBC encryption CRC-16 integrity Titanium era (2004)

Three-Port Architecture

The EQ Titanium client connects to three separate UDP ports in sequence. Each represents a distinct phase of the connection lifecycle. The client tears down one session before opening the next.

Session Protocol Layer

All application packets are wrapped in a session-layer protocol. This handles connection setup, sequencing, acknowledgment, fragmentation, and integrity checks. Session opcodes use big-endian byte order.

0x0001 SessionRequest 14 bytes
BE [opcode:2] [session_id:4] [protocol_version:4] [padding:4]

Client sends to initiate a new session. Session ID is client-chosen.

0x0002 SessionResponse 17 bytes
BE [opcode:2] [session_id:4] [key:4] [crc_bytes:1] [format:1] [pad:1] [max_len:2] [pad:2]

Server responds with CRC config. crc_bytes=0 during login, crc_bytes=2 during world/zone.

0x0003 Combined variable
BE [opcode:2] [len1:1] [subpacket1...] [len2:1] [subpacket2...] ...

Multiple subpackets in one UDP datagram. Length-prefixed with a single byte per subpacket.

0x0009 Packet (Sequenced) variable
BE [opcode:2] [sequence:2] LE [app_opcode:2] [app_data...]

Single application-layer packet with sequence number. App opcode is little-endian (unlike session opcodes).

0x000D Fragment variable
First: BE [opcode:2] [seq:2] [total_size:4] [data...] Next: BE [opcode:2] [seq:2] [data...]

First fragment includes total reassembled size (4 bytes BE). Subsequent fragments are data-only with incrementing sequence.

0x0015 Ack 6 bytes
BE [opcode:2] [seq:2] [crc:2]

Acknowledges receipt of all packets up to and including the given sequence number.

0x0006 KeepAlive 4 bytes

Bidirectional heartbeat. Sent every ~8 seconds during idle periods.

0x0007 / 0x0008 SessionStat Req/Resp 42 bytes

Statistics exchange for round-trip time estimation and packet loss tracking.

Protocol Notes

i

Byte Order Split

Session-layer opcodes and fields use big-endian. Application-layer opcodes use little-endian. This split is a historical artifact of the SOE network library vs. the EQ game code.

i

CRC Behavior

CRC (2 bytes) is appended to most packets after session establishment. During the login phase, the server responds with crc_bytes=0 so no CRC is used. During world and zone phases, crc_bytes=2 enables CRC-16 on all sequenced packets.

i

Fragmentation Threshold

Application payloads exceeding ~512 bytes are broken into fragments (opcode 0x000D). The first fragment carries the total reassembled size. The receiver buffers fragments by sequence number until complete. Both GuildsList (96KB) and SendCharInfo (1.7KB) use fragmentation in practice.

Proactive Server Sequence

After receiving OP_SendLoginInfo, the server sends these packets automatically in this exact order. The client does NOT request them.

1
OP_GuildsList (0x6957) 96,066 bytes (fragmented)

Full guild name table. EQEmu sends 96KB; mostly zeros for empty guild slots.

2
OP_LogServer (0x0FA6) 264 bytes

Server shortname ("ADIF") at offset 32. Contains PVP/FV flags.

3
OP_EnterWorld (0x7CBA) auto-login character name (or empty)

Server-to-client direction: suggests last-played character for auto-login.

4
OP_PostEnterWorld (0x52A4) 0 bytes

Empty packet, just the opcode. Signals post-enter-world phase to client.

5
OP_ExpansionInfo (0x04EC) 4 bytes

Expansion bitmask. Capture shows 0x000003FF (1023).

6
OP_SendCharInfo (0x4513) 1,704 bytes (fragmented)

CharacterSelect_Struct with 10 character slots. Fragmented across ~4 packets.

Client sends OP_ApproveWorld after receiving the above.

After user selects a character and hits "Enter World":

CLIENT OP_EnterWorld (0x7CBA) character name
SERVER OP_MOTD (0x024D) message of the day
SERVER OP_ZoneServerInfo (0x61B6) zone IP + port
BOTH SessionDisconnect world session ends

Bridge Comparison

Differences between the EQEmu reference server and the ADIF Rust bridge, observed through packet capture comparison. Color-coded by severity: matches minor difference significant difference.

1
GuildsList Size Mismatch Minor
OP_GuildsList (0x6957)

EQEmu Reference

Sends 96,066 bytes (fragmented across ~190 packets). Full guild name table, mostly zeros for empty slots.

ADIF Bridge

Sends 4 bytes: [0u8; 4] (zero guild count).

Probably NOT the black screen cause. An empty guild list is a valid response -- it just means no guilds exist on the server.

2
Premature OP_ApproveWorld Critical
OP_ApproveWorld (0x3C25)

EQEmu Reference (Correct)

  1. Server sends: GuildsList, LogServer, EnterWorld, PostEnterWorld, ExpansionInfo, SendCharInfo
  2. Client sends OP_ApproveWorld
  3. Server acknowledges

ADIF Bridge (Incorrect)

  1. Server sends: GuildsList
  2. Server sends OP_ApproveWorld PROACTIVELY
  3. Then sends LogServer, etc. after OP_SendLoginInfo

THIS COULD BE THE BUG

The bridge sends OP_ApproveWorld from the server side before the client sends it. In the EQEmu reference, OP_ApproveWorld is a CLIENT-initiated packet that the server WAITS for after delivering the character list. Sending it prematurely from the server may confuse the client's state machine and cause the black screen on character select.

3
ExpansionInfo Bitmask Difference Minor
OP_ExpansionInfo (0x04EC)

EQEmu Reference

0x03FF (1023) -- 10 expansions enabled

ADIF Bridge

0x7FFF (32767) -- 15 expansions enabled

Different but probably not causing the black screen. The Titanium client may ignore unknown expansion bits.

4
LogServer Flags Minor
OP_LogServer (0x0FA6)

Bridge sends 264 bytes correctly. The enable_pvp and enable_FV flags may differ from the reference server, but the struct size matches.

5
CharacterSelect_Struct Size Match
OP_SendCharInfo (0x4513)

Both EQEmu and the bridge send exactly 1,704 bytes. Struct size is correct.

6
Fragmentation Handling Minor

EQEmu fragments large packets (GuildsList at 96KB, SendCharInfo at 1.7KB). The bridge should do the same for any application payload exceeding ~512 bytes. Failure to fragment may cause the client to drop oversized UDP datagrams.

Investigation Priority

P0

Fix OP_ApproveWorld direction

Remove the server-side proactive send of OP_ApproveWorld. Wait for the client to send it after receiving the character list, then acknowledge it. This is the most likely cause of the black screen.

P1

Verify packet ordering matches EQEmu exactly

GuildsList, LogServer, EnterWorld, PostEnterWorld, ExpansionInfo, SendCharInfo -- in that exact order, all before the client sends anything after OP_SendLoginInfo.

P2

Match ExpansionInfo bitmask

Use 0x03FF instead of 0x7FFF to exactly match the reference server.

P3

Ensure proper fragmentation for large payloads

Any app payload > 512 bytes must be sent as fragments (0x000D). This is handled at the session layer.
