ADIF Architecture Study

How EverQuest Works

A technical deep-dive into the architecture of a 1999 MMORPG that pioneered the genre. From cluster topology to combat formulas — how 500,000 concurrent players were served by technology older than most of its players.

231 database tables 500+ zones ~100k NPCs Released March 16, 1999

The Cluster Model

EverQuest runs as a cluster of independent processes. There is no single "game server" — instead, five types of processes cooperate, connected through the world server and a shared database. This design was radical in 1999 and remains the foundation for most MMO server architectures today.

Login Server

loginserver.exe

The front door. Handles account authentication and presents the server list. Once you pick a server, it hands you off to that server's world process and its job is done.

Listens on port 5998 (client) + 5999 (world servers)
Tables: login_accounts, login_world_servers, login_api_tokens

World Server

world.exe

The coordinator. Manages character selection, decides which zone process handles each player, routes cross-zone messages (tells, /who, guild chat), and orchestrates zone transfers.

Listens on port 9000 (zone servers) + 9080 (web API)
Launches zone.exe processes on demand (ports 7000–7400)

Zone Server

zone.exe (one per active zone)

Where the game actually happens. Each active zone runs as its own OS process. Handles everything inside that zone: NPC AI, combat, spells, movement validation, loot drops, quest scripting, trade skills. If a zone crashes, only that zone's players are affected.

Loads geometry files: .map (collision), .wtr (water), .nav (pathfinding)
Spawns NPCs from database, runs AI tick loop at ~10 Hz
Communicates only with: world server, database, connected clients

Database

MySQL / PostgreSQL

The single source of truth for all persistent state. Characters, items, NPCs, spawn points, loot tables, quest data, faction standings — if it survives a server restart, it lives here. EQEmu uses 231 tables.

Character saves happen on zone exit, periodic timer, or /save
NPC/spawn/item/loot data loaded once at zone boot, cached in memory
NOT used for: position updates, combat ticks, ephemeral state

Why This Design Works

Fault Isolation

A zone crash only affects players in that zone. The world server detects the crash, marks the zone as down, and can relaunch it. Other zones keep running.

Horizontal Scaling

Zone processes can run on different machines. A popular expansion launch can spread new zones across additional hardware without touching existing infrastructure.

Simplicity

Each zone process only thinks about one zone. No global lock contention, no shared memory between zones. The world server is the only coordination point.

From Player to Packet

Every moment a player experiences in EverQuest has a chain of systems making it happen. Here's what's behind the curtain.

STEP 1

You launch EQ and log in

What you see

The patcher window, then the login screen. You enter your username and password. A server list appears — you pick your server.

What happens

eqgame.exe patchme connects to the login server on port 5998.

Your password is hashed and checked against the login_accounts table.

The login server queries all registered world servers for their status and player count, then sends you the list.

When you pick a server, the login server gives you a session token and tells you the world server's IP and port. Your client disconnects from login and connects to world.

STEP 2

Character select screen

What you see

Your characters standing in a row. Their names, levels, classes, and zones are displayed. You can create a new character or enter the world.

What happens

The world server validates your session token, then queries character_data for all characters on this account (106 columns per character).

It sends back a packet with each character's appearance data (race, class, level, equipment visual IDs) so the client can render the 3D models.

Character creation writes to character_data, character_bind (spawn point), character_skills, and starting_items.

STEP 3

Entering the world — the zone handoff

What you see

"Loading, please wait..." appears. Then the zone loads — terrain, buildings, NPCs pop in. You're standing where you last logged out (or at your bind point).

What happens

The world server checks if a zone process is already running for your destination zone.

If not: it launches a new zone.exe. The zone process boots in ~6 seconds: loads .map/.wtr/.nav geometry from disk, queries all spawn points from the database, creates NPC instances, loads quests.

The world server sends your client the zone process's IP and port. Your client disconnects from world and connects directly to the zone.

The zone process loads your full character from the database (inventory, spells, skills, buffs, faction, AAs) and sends you the spawn packets for every entity in the zone.

// Zone boot sequence (simplified)
1. Load Map (collision AABB tree from .map file)
2. Load WaterMap (liquid regions from .wtr file)
3. Load PathFinder (NavMesh from .nav file)
4. Query spawn2 table → create NPC instances
5. Load quest scripts (Perl/Lua)
6. Report "ready" to world server
7. Accept client connections
STEP 4

Moving around the zone

What you see

WASD movement, smooth terrain, other players running around, NPCs patrolling on paths. You can swim in water and fall off cliffs.

What happens

The client sends position updates to the zone server. The server validates them against the Map collision data (can you actually stand there? is there a wall in the way?).

The server broadcasts your position to all other clients in the zone. Each client interpolates between position updates to make movement look smooth.

NPCs follow patrol paths from the grid / grid_entries tables. Their pathfinding uses the NavMesh to navigate around obstacles.

The WaterMap tells the server if you're swimming (applies swim speed, drowning timer, underwater breathing checks).

STEP 5

Fighting a mob

What you see

You target a Froglok and auto-attack. Damage numbers fly. The mob hits back. You cast a spell — it resists! You try again — it lands. The mob dies, a corpse appears with loot.

What happens

Client sends "I want to attack this target." The server decides everything else.

Every ~2 seconds (weapon delay), the server runs the melee round: calculates hit/miss using ATK vs AC, rolls damage, applies damage shields, checks for procs, ripostes, dodges.

Spells: the server checks mana cost, casting time, range, line of sight (using the Map), resist check (level + magic resistance), then applies the spell effect.

NPC AI runs on a tick (~1.5s): evaluates aggro list, picks target, decides to flee at low HP, calls for help if scripted.

On death: server rolls loot from loottable → lootdrop → lootdrop_entries, creates a corpse entity, grants XP split to the group, fires quest EVENT_DEATH.

STEP 6

Zoning — crossing between zones

What you see

You walk into a tunnel or zone line. "Loading, please wait..." — the new zone loads. You appear on the other side.

What happens

Your character hits a zone point (from the zone_points table) — a bounding box at the zone boundary with a destination zone and coordinates.

The zone server saves your character to the database, removes you from its entity list, and tells the world server "this player wants to go to Innothule."

The world server checks if a zone process is running for Innothule. If not, it launches one. It gives your client the new zone's address.

Your client disconnects from zone A, connects to zone B. Zone B loads your character fresh from the database and spawns you at the destination coordinates.

The two zone servers never communicate directly. The world server is the relay.

STEP 7

Logging out (or dying)

What you see

You camp out (30-second timer in a dangerous zone, instant in a city). Or you die — your corpse drops with all your gear and you respawn at your bind point, naked.

What happens

Camp out: Zone server saves your full character state to the database — position, HP, mana, buffs, inventory, everything. Removes you from the entity list. Tells world server you're offline.

Death: The server creates a character_corpses entry with all your equipment. Your character_data gets a zone change to your bind point. XP penalty applied. Corpse has a decay timer.

If the zone is empty after you leave, the zone process stays alive for a while, then shuts down to free resources.

Networking & Protocol

UDP • opcodes • packet structure

EQ uses UDP with a custom reliability layer on top. This was unusual in 1999 — most games used TCP. The custom layer gives EQ control over retransmission, packet ordering, and fragmentation without TCP's head-of-line blocking.

Packet Structure

[2 bytes] Opcode — identifies the packet type
[N bytes] Payload — serialized struct data
[2 bytes] CRC — integrity check

Opcodes are version-specific — different client versions use different opcode tables. The server uses an opcode translation layer to support multiple clients.

Key Packet Types

OP_PlayerProfile — full character data (sent on zone entry)
OP_ClientUpdate — position/heading updates (~10/sec)
OP_Damage — combat damage numbers
OP_SpawnAppearance — entity visual changes
OP_NewSpawn — new entity enters zone
OP_ZoneChange — zone transfer request

Security note: The original EQ protocol sends structs almost directly on the wire. Field offsets are client-version-dependent. This is why EQEmu needs different "patch" files for different client versions — each translates between the server's internal structs and the client's expected wire format.

Zone Geometry

Map • WaterMap • PathFinder • S3D

Zone geometry is completely decoupled from game logic through three narrow interfaces. The server and client use entirely different geometry data — the server never sends mesh data, and the client never does collision.

Map (.map files)

Collision detection and line of sight. A spatial tree (AABB) of triangles extracted from the zone mesh.

• Ground detection (Z coordinate for any X,Y)
• Line-of-sight checks (can NPC see player?)
• Zone boundary enforcement
• ~2-10 MB per zone

WaterMap (.wtr files)

Liquid region detection. Defines volumes of water, lava, slime, and other fluids.

• Is the player swimming?
• Lava damage ticks
• Underwater breathing timers
• Fish-spawning regions (for fishing)

PathFinder (.nav files)

NPC navigation mesh. Built with Recast/Detour, the same library used by Unreal Engine.

• NPC pathfinding (A* on NavMesh)
• Terrain cost calculations
• Avoidance of obstacles
• Patrol waypoint routing

Client-Side: S3D Archives

The client has its own copy of zone geometry in proprietary S3D archives — these contain the actual 3D meshes, textures, and vertex data for rendering. The server never touches these files. This separation means the server could use completely different geometry (voxels, procedural terrain) as long as the three server-side interfaces are satisfied.

ADIF insight: This decoupling is the key architectural insight that makes pluggable zone geometry possible. A zone running voxel geometry just needs to provide the same Map/WaterMap/PathFinder interfaces. The zone server doesn't care how the geometry was generated.

NPC AI & Behavior

Aggro • hate lists • pathing • scripts

NPC behavior is driven by a simple but effective priority system. There's no behavior tree or GOAP — it's a state machine with an aggro list.

The Hate List

Every NPC maintains a sorted list of entities it hates. The entity with the most hate is the current target. Hate is generated by:

• Damage dealt → proportional hate
• Healing → hate on the healer from all mobs fighting the healed target
• Taunt → large hate spike + temporary lock
• Spell aggro → base hate per spell, modified by type
• Proximity → initial aggro when entering NPC's aggro radius

NPC States

Idle: Standing still or patrolling along grid waypoints
Aggro scan: Checking for hated entities within aggro_range (faction-based)
Combat: Attacking highest-hate target, casting spells from npc_spells list
Flee: Running away at low HP (configurable threshold)
Return: Walking back to spawn point after losing all targets

Quest Scripting (Perl/Lua)

NPCs can have custom behavior through quest scripts. Scripts are event-driven — the zone server fires events and the script responds:

EVENT_SAY — player says something near the NPC
EVENT_TRADE — player hands items to the NPC
EVENT_DEATH — NPC dies
EVENT_SPAWN — NPC first appears in the zone
EVENT_COMBAT — NPC enters/leaves combat
EVENT_TIMER — scheduled callback fires

Combat Formulas

Melee • spells • resists • mitigation

All combat resolution happens server-side. The client only displays results. The core loop runs every weapon-delay interval (~2-4 seconds for melee).

// Simplified melee attack resolution
1. Check range — is target within melee distance?
2. Avoidance roll — riposte → dodge → parry → block (each has a % chance)
3. Hit/miss roll — attacker ATK vs defender AC, modified by level difference
4. Damage roll — weapon min/max damage, modified by STR bonus
5. Mitigation — defender's AC reduces damage (with diminishing returns)
6. Critical hit — % chance to multiply damage (based on DEX + AAs)
7. Damage shields — if target has a DS buff, attacker takes reflected damage
8. Proc check — weapon procs fire at a rate based on weapon speed
// Spell resist check
resist_chance = target_resist - (caster_level - target_level) * 2
roll = random(0, 200)
if roll < resist_chance → full resist (no effect)
else if roll < resist_chance * 2 → partial resist (reduced duration/damage)
else → spell lands fully

These formulas evolved over 25+ years of patches. The emulator community reverse-engineered them by comparing observed behavior against Sony's servers. Some formulas are still debated.

Loot System

loottable → lootdrop → items • global loot

Loot is a three-tier random system. When an NPC dies, the server walks the chain:

npc_types.loottable_id

Points to a loottable

→

loottable_entries

Links to N lootdrops with probability, multiplier, and drop limit

→

lootdrop_entries

Pool of items with individual drop chances, multipliers, and min/max quantities

→

items

The actual item definitions (285 columns each)

Additionally, the global_loot table can inject items into any NPC matching criteria (level range, zone, body type). This powers world drops — items that can drop from any mob of a certain type.

Faction System

Race/class modifiers • NPC reactions • faction hits

Faction determines how NPCs react to you. It's a numeric standing that changes based on your actions — killing an NPC lowers your standing with their faction and may raise it with rival factions.

// Faction check when player approaches NPC
base_faction = faction_base_data[npc.primary_faction]
race_mod = npc_faction_entries[player.race]
class_mod = npc_faction_entries[player.class]
deity_mod = npc_faction_entries[player.deity]
personal = faction_values[player.id, faction_id] // earned through kills
total = base_faction + race_mod + class_mod + deity_mod + personal
// Thresholds:
// > 750 = Ally, > 500 = Warmly, > 100 = Kindly, > 0 = Amiable
// > -100 = Indifferent, > -500 = Apprehensive, > -750 = Threatening
// ≤ -750 = Ready to Attack (KOS)

This is why a Dark Elf Necromancer is kill-on-sight in Qeynos (race + class penalties) but could theoretically grind enough faction kills to walk freely. Race/class/deity are fixed modifiers; personal standing is earned.

Spawn System

spawn2 → spawngroup → spawnentry → npc_types

The spawn system is another chain like loot. A point in the world is linked to a group of possible NPCs, each with a spawn chance.

spawn2

A point: X, Y, Z, heading, respawn timer, zone, enabled flag

→

spawngroup

A named group with spawn limit and roaming distance

→

spawnentry

Links NPCs to the group with spawn chance (0-100%)

→

npc_types

The NPC template (131 columns of stats/appearance)

This allows "placeholder" spawns — one spawn point can produce different NPCs each time. Classic EQ raid targets like Nagafen had a placeholder: most of the time a trash mob spawned, but occasionally the dragon appeared.
