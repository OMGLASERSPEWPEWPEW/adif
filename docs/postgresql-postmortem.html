The Starting Point

What We Started With

  • EQEmu server — a C++ MMORPG emulator built entirely around MySQL. Every query, every schema assumption, every string escape function: MySQL.
  • 250 repository files generating MySQL-specific SQL — REPLACE INTO, FROM_UNIXTIME, backtick quoting, AUTO_INCREMENT, MySQL-flavored type casts everywhere.
  • akk-stack deployment running MariaDB 10.11 with a live game database — 231 tables, 1.8M+ rows of content data (NPCs, items, spells, zones).
  • Runtime translation layer — a RewriteQuery() function that tried to patch MySQL syntax at runtime. Fragile and incomplete.

What We Wanted

  • PostgreSQL-native — no translation layers, no compatibility shims. Real PostgreSQL queries with proper types, ON CONFLICT, RETURNING, and JSONB.
  • Full schema parity — every table from the MySQL database present in PostgreSQL with matching columns, types, and primary keys.
  • Server boots and runs — login, character creation, zone entry, NPC spawning. The full gameplay loop on PostgreSQL.
  • Foundation for ADIF — PostgreSQL gives us JSONB for flexible schemas, better type safety, and a modern database to build the real game on.

Hardest Bugs

1

FindReplace("", "-") Infinite Loop

NPC name formatting called FindReplace with an empty search string, causing an infinite loop that hung zone boot completely. The zone process would start, load map data, connect to the database, begin spawning NPCs — and then freeze. No error, no crash, no log output. Just silence.

// The culprit: empty string search = infinite loop
FindReplace(npc_name, "", "-");

// Found by adding timing instrumentation to Zone::Init
// Each init phase logged its duration — "SpawnNPCs" never completed
2

RETURNING id Across 250 Files

PostgreSQL INSERT doesn't return last_insert_id() like MySQL. Every InsertOne method across 250 auto-generated repository files needed RETURNING id appended to its INSERT statement. Without it, character creation would succeed in the database but the server would get back id=0, causing every subsequent operation to fail silently.

// MySQL: implicit last_insert_id()
INSERT INTO characters (name, ...) VALUES (...);
auto id = database.LastInsertId();

// PostgreSQL: explicit RETURNING
INSERT INTO characters (name, ...) VALUES (...) RETURNING id;
auto id = row[0]; // returned in the result set
3

Boolean DEFAULT Cast Failures

PostgreSQL refuses to ALTER COLUMN from BOOLEAN to SMALLINT if the column has a boolean DEFAULT value. The migration would fail with a type cast error. The fix required a three-step dance for every affected column: drop the default, alter the type, then re-set the default as an integer.

-- This fails:
ALTER COLUMN is_active TYPE SMALLINT USING is_active::integer;
-- ERROR: default for column "is_active" cannot be cast to SMALLINT

-- Three-step fix:
ALTER COLUMN is_active DROP DEFAULT,
ALTER COLUMN is_active TYPE SMALLINT USING is_active::integer,
ALTER COLUMN is_active SET DEFAULT 0;
4

Account ID Zero

The migrated account had id=0, but EQEmu treats id=0 as "not found." First login worked — the login server found the account by username. But every subsequent operation queried by account ID, got back zero, and failed silently. Character list empty. Character creation succeeds but doesn't attach to the account. A phantom user in a phantom world.

// EQEmu pattern throughout the codebase:
auto account_id = GetAccountID(username);
if (account_id == 0) {
  return "Account not found";
}

// Fix: PostgreSQL SERIAL starts at 1, but migrated data kept id=0
// Solution: UPDATE account SET id = nextval('account_id_seq');

The Comparison Script

The permanent ground truth for schema parity. scripts/compare-pg-to-mysql.py connects to both MariaDB (port 3306, database peq) and PostgreSQL (port 5433, database adif), enumerates every table and column, and produces a detailed comparison report. It's rerunnable at any time — run it after any migration to verify nothing regressed.

What it checks:

Table presence in both databases
Column names and order
Column types (with MySQL → PG type mapping)
Primary key definitions
Missing columns in either direction
Extra PG-only tables (ADIF additions)
