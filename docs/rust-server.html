Technology Stack

🦀
Rust 1.96
Memory safety without GC pauses. Ownership model prevents data races between ECS systems. Zero-cost abstractions for hot game loop paths.
⚡
tokio
Async runtime for TCP networking, database queries, and I/O. Multi-threaded work-stealing scheduler. The standard for async Rust servers.
🎮
bevy_ecs 0.16
Standalone ECS — no full Bevy engine. Cache-friendly archetypal storage. Systems run as plain functions querying component tuples. Already used in production game servers.
🗄️
sqlx + PostgreSQL
Async PostgreSQL driver. Compile-time query checking catches schema drift at build time. Port 5433, database "adif", 231 tables at 100% EQEmu parity.
📦
prost + protoc 28.3
Protobuf serialization via code generation at build time. All 17 proto files (173 client-server + 77 IPC messages) compile to type-safe Rust structs.
📊
tracing
Structured, context-aware logging. Spans for request tracing. Filterable by module (adif_zone, adif_common). Production-ready from day one.

The original plan was C#/.NET. Switched to Rust before any server code was written (June 27, 2026). The deciding factor: an MMORPG zone server runs hundreds of entities at 31 Hz with hard tick deadlines — GC pauses are the #1 risk to game feel.

Pluggable Zone Geometry

Zone geometry is decoupled from game logic through three trait interfaces. Different zones can use different geometry backends (mesh, voxel, procedural) and zone-to-zone transitions still work seamlessly.

trait Map
Collision detection, ground height (best_z), line-of-sight queries. The physics backbone.
EQ format: .map (AABB tree)
trait WaterMap
Liquid region detection — water, lava, slime. Affects movement speed, breathing, damage.
EQ format: .wtr (region volumes)
trait Pathfinder
NPC navigation with terrain costs. A* over navmesh. Drives patrol routes and chase behavior.
EQ format: .nav (Recast/Detour)
Phase 11 deliverable. Null implementations (flat plane) used until then.

Architecture Decisions

Key choices made during Milestone 3 design, with rationale and alternatives considered.

1

bevy_ecs standalone, not full Bevy

We use bevy_ecs as a library crate — no Bevy App, no rendering pipeline, no asset system. Just the ECS: World, Components, Systems, Queries.
Why
An MMORPG server is headless — it never renders anything. Pulling in the full Bevy engine adds 200+ transitive crates (wgpu, winit, gilrs, etc.) that we'd never use. bevy_ecs alone gives us archetypal storage, parallel system scheduling, and change detection.
Alternatives Considered
hecs — simpler but lacks system scheduling, change detection, and resources. specs — older, less maintained. legion — good but smaller community. bevy_ecs wins on ecosystem, documentation, and the option to adopt more of Bevy later if needed.
2

sqlx over diesel / sea-orm

Raw SQL with sqlx, no ORM. Queries are hand-written strings checked at compile time against the real database schema.
Why
MMO data is relational but access patterns are highly specialized (hot paths: NPC lookups, spawn resolution, character saves). An ORM's object-graph assumptions fight the ECS data model. sqlx gives full control over query plans while catching typos and schema drift at build time.
Alternatives Considered
Diesel — compile-time safety but invasive DSL, fight over schema ownership with migrations. SeaORM — ActiveRecord pattern doesn't fit ECS. Raw SQL with sqlx is the thinnest possible layer over PostgreSQL.
3

prost over tonic / protobuf-native

prost for serialization only. No gRPC layer — we use raw TCP with length-delimited protobuf frames. The existing buf toolchain handles linting and breaking-change detection.
Why
Game servers need precise control over the wire format. gRPC's HTTP/2 framing adds overhead and complexity for no benefit — game packets are small, frequent, and don't need HTTP semantics. prost generates plain Rust structs that encode/decode protobuf without any networking assumptions.
Alternatives Considered
tonic — gRPC is wrong for game traffic (latency, framing overhead). protobuf (Google's Rust bindings) — heavier, requires protoc plugin, less idiomatic. prost is the community standard for protobuf in Rust.
4

Entity ID allocator over raw ECS Entity

A u32 entity_id is allocated per game entity and sent over the wire. bevy_ecs::Entity is internal only. The allocator maintains a bidirectional HashMap.
Why
bevy_ecs::Entity is a 64-bit value with an internal generation counter — it's an implementation detail that shouldn't leak into the protocol. A u32 entity_id is smaller on the wire (4 bytes vs 8), stable across server restarts, and matches EQ's entity numbering convention. ID 0 is reserved (skipped on wrap).
Alternatives Considered
Send Entity directly — leaks ECS internals, doubles wire cost, breaks if we switch ECS. UUID — 16 bytes per entity × thousands of position updates per second = unnecessary bandwidth. u32 gives 4 billion IDs before wrap.
5

Components mirror proto fields, not EQEmu classes

ECS components are designed to map cleanly to protobuf Spawn message fields — not to replicate EQEmu's C++ class hierarchy (Mob → NPC → Client → Corpse).
Why
EQEmu's inheritance hierarchy (Entity → Mob → NPC / Client) bakes entity type into the class structure. ECS replaces this with composition: an entity is Player because it has a ClientSession component, not because it inherits from a Player class. This makes the Spawn conversion trivial — each component maps to a proto field group.
Design Principle
The proto Spawn message is the canonical "what does an entity look like to the outside world" definition. If a field exists in Spawn, there's a component for it. If a component doesn't map to Spawn, it's internal state (e.g., NpcTemplate.npc_type_id, ClientSession).
