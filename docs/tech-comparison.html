Tech Comparison Server Proof Decisions

EverQuest Tech (1999) vs Modern Approaches

What EQ did, why it worked, and where ADIF can do better — plus a rethought server-side proof-of-concept plan.

KEEP — still the right call UPGRADE — better options exist RETHINK — fundamentally different approach NEW — didn't exist in EQ

1. Networking

UPGRADE

EQ / EQMacEmu (1999)

  • • Custom reliability protocol over raw UDP — hand-built sequencing, acknowledgments, fragmentation (OP_SessionRequest, OP_Ack, OP_Fragment)
  • • libuv event loop for async I/O (uv_udp_t socket)
  • • Position updates packed into ~11 bytes with bit-shifting to save bandwidth on dial-up
  • • No encryption, no modern auth — session handshake only
  • • Connection states managed manually: CLIENT_CONNECTING → CLIENT_AUTH_RECEIVED → CLIENT_CONNECTED → CLIENT_LINKDEAD

Modern Options

  • • ENet — proven UDP reliability library used by many game servers. Handles sequencing, fragmentation, channels. Drop-in replacement for EQ's hand-rolled protocol.
  • • QUIC (HTTP/3) — built-in encryption + reliable streams over UDP. Modern but more complex.
  • • WebTransport / WebRTC — if going browser-based client. WebRTC data channels give you UDP-like unreliable + reliable channels.
  • • Interest management — only send updates for entities near the player. EQ sent everything in the zone. Huge bandwidth savings in crowded zones.
  • • Delta compression — only send what changed since last ack'd state. Beats EQ's fixed-format position packets.
ADIF opportunity: Don't hand-roll a reliability protocol. Use ENet (if native client) or WebRTC data channels (if browser). Add interest management from the start — EQ's "broadcast everything to everyone in zone" doesn't scale past ~100 players per zone.

2. Serialization / Packet Format

UPGRADE

EQ / EQMacEmu

  • • Hand-rolled C structs with #pragma pack(1) — raw binary memcpy, ~2,900 lines of struct definitions
  • • Tightly coupled to client version — different struct layouts per client era
  • • No schema evolution — changing a field breaks everything
  • • Newer code uses cereal (C++ template serialization) for some packets
  • • Opcode-based dispatch: 1-2 byte opcode → handler function

Modern Options

  • • FlatBuffers — zero-copy deserialization, schema evolution, cross-language. Ideal for game packets (fastest option).
  • • Protocol Buffers — schema evolution, code generation, well-tooled. Slightly more overhead than FlatBuffers but more ergonomic.
  • • MessagePack — binary JSON. Flexible but no schema validation.
  • • Cap'n Proto — zero-copy like FlatBuffers, with RPC built in.
ADIF opportunity: Use FlatBuffers or Protobuf with a shared schema file that generates code for both server and client. Schema evolution means we can add fields without breaking old clients. The opcode dispatch pattern is still solid — keep that, just use a real serialization format for the payloads.

3. Entity / Game Object Model

RETHINK

EQ / EQMacEmu

  • • Deep inheritance hierarchy:
    Entity → Mob → Client
    Entity → Mob → NPC
    Entity → Corpse / Object / Door / Trap / Beacon
  • • Mob class is massive — HP, level, race, class, position, equipment, special abilities all in one class
  • • Type checking via virtual methods: IsClient(), IsNPC(), CastToClient()
  • • EntityList stores everything in hash maps by type
  • • Sequential processing: entity_list.MobProcess(), .DoorProcess(), etc.

Modern: Entity Component System (ECS)

  • • Composition over inheritance — entities are just IDs. Behavior comes from attached components.
  • • Position, Health, AIBehavior, Inventory are all separate components
  • • Systems process entities by component query: "give me everything with Position + Health + AI"
  • • Cache-friendly data layout (structs of arrays, not arrays of structs)
  • • Libraries: bevy_ecs (Rust), flecs (C/C++), bitECS (JS), Unity DOTS
  • • Trivially parallelizable — systems that touch different components run concurrently
ADIF opportunity: ECS is the biggest architectural win available. EQ's Mob class doing everything made sense in 1999 C++ but creates a maintenance nightmare. With ECS, adding a new behavior (e.g., "this NPC can be mounted") is just attaching a component — no class hierarchy changes. It also makes the zone server naturally parallelizable.

4. Database & Persistence

UPGRADE

EQ / EQMacEmu

  • • MySQL/MariaDB via raw C API (libmysql)
  • • Mostly raw SQL strings: QueryDatabase(fmt::format(...))
  • • Auto-generated repository classes for common CRUD (InsertOne, FindOne, GetWhere)
  • • Single DB connection per process with mutex locking
  • • Character saves are periodic bulk writes — lose data on crash
  • • All game content (spawns, items, quests) stored as rows

Modern Options

  • • PostgreSQL — JSONB columns for flexible schemas, better concurrency than MySQL
  • • Connection pooling — PgBouncer / built-in pools instead of one-connection-per-process
  • • Event sourcing — log every state change, reconstruct state from events. Perfect for MMO audit trails, rollbacks, and GM tools
  • • Redis/Valkey — in-memory cache for hot data (who's online, zone populations, auction house)
  • • SQLite — embedded DB for zone-local data that doesn't need global persistence
ADIF opportunity: PostgreSQL as the primary store. Add Redis for real-time state (who's in which zone, chat presence). Consider write-ahead logging for character state so crashes don't lose progress. The EQMacEmu repository pattern is actually decent — keep the concept, just use a proper query builder or ORM.

5. Threading & Concurrency

UPGRADE

EQ / EQMacEmu

  • • Single-threaded zone server — one main loop processes everything sequentially
  • • libuv event loop for async I/O but game logic is synchronous
  • • Processing order each tick: Groups → Doors → Objects → Corpses → Mobs → Beacons → Encounters → Zone → Quests
  • • IPC via memory-mapped files + named mutexes between zone and world processes
  • • DB queries block the game loop (mutex-locked single connection)

Modern Options

  • • Async I/O with thread pool — Tokio (Rust), .NET async, or libuv workers for DB/disk
  • • Actor model — each zone, NPC group, or system is an actor with message passing. Natural fit for zone processes.
  • • ECS parallelism — systems that touch different components run on different threads automatically
  • • gRPC / message queues — replace memory-mapped IPC with proper service communication (NATS, RabbitMQ, or just gRPC)
  • • Async DB: non-blocking queries so the game loop never stalls
ADIF opportunity: Single-threaded game logic is actually fine for small zones — it avoids concurrency bugs. But DB queries must be async (never block the tick). Use gRPC or a message queue between zone ↔ world instead of shared memory files. If using ECS, get parallelism for free within a zone.

6. Collision & Spatial Data

RETHINK

EQ / EQMacEmu

  • • AABB tree over triangle soup — zone mesh converted to triangles, stored in axis-aligned bounding box hierarchy
  • • Raycasting for FindBestZ() (ground detection) and CheckLoS() (line of sight)
  • • Preprocessing pipeline: S3D mesh → AZONE converter → .map binary file
  • • Water regions stored separately in .wtr files
  • • Static geometry only — nothing moves or changes

Voxel-Native Approach

  • • Collision is trivial with voxels — just check if the target voxel is solid. No raycasting needed for basic checks.
  • • Octree for sparse voxel storage — O(log n) spatial queries, natural LOD
  • • Voxel ray marching for line-of-sight — step through the grid, faster than triangle intersection for blocky worlds
  • • Water/lava/material is just a voxel property — no separate water map needed
  • • Dynamic terrain — change a voxel, collision updates instantly. No re-baking.
  • • Chunk-based streaming: load/unload 16³ or 32³ chunks on demand
ADIF opportunity: This is where voxels shine brightest on the server side. EQ needed a complex preprocessing pipeline (S3D → AZONE → .map) and a separate water map. With voxels, collision is a voxel lookup, water is a material type, and terrain can be modified at runtime. The three separate files (.map, .wtr, .nav) collapse into one voxel volume + a NavMesh.

7. Pathfinding & NPC Navigation

KEEP (mostly)

EQ / EQMacEmu

  • • Original: Grid waypoints — hand-placed patrol paths with pause times. Simple but rigid.
  • • Upgraded: Recast/Detour NavMesh — industry-standard navigation mesh with terrain cost weighting
  • • Terrain costs: Normal=1.0, Water=3.0, Lava=5.0, Ice=4.0, Disabled=impassable
  • • Three pathfinder implementations: NavMesh, Waypoint (legacy), Null (direct line)
  • • NavMesh is prebaked — if terrain changes, you rebuild the whole zone's .nav file

Modern Additions

  • • Recast/Detour is still the gold standard — used in Unity, Unreal, and most AAA games. No need to replace it.
  • • Dynamic NavMesh updates — Detour supports tile-based updates for terrain changes without full rebuild
  • • Voxel-native pathfinding — A* directly on the voxel grid for simple cases. Jump Point Search for optimization.
  • • Hierarchical pathfinding (HPA*) — chunk-level paths first, detailed paths within chunks. Great for large voxel worlds.
  • • Flow fields — precompute movement direction per cell. Efficient when many NPCs go to the same target.
ADIF opportunity: Keep Recast/Detour for the NavMesh — it already works great with voxel input (voxels are ideal Recast input since they're grid-aligned). Use tile-based NavMesh so terrain changes only rebuild affected chunks. Grid waypoints for scripted patrols are still fine — don't over-engineer those.

8. Quest & NPC Scripting

UPGRADE

EQ / EQMacEmu

  • • Lua via luabind for quest scripting (Perl legacy also supported)
  • • Event-driven: EventNPC(), EventPlayer(), EventItem(), EventSpell()
  • • QuestManager coordinates execution with timer support
  • • Rich API: say(), emote(), spawn2(), castspell(), addloot()
  • • Scripts loaded per-NPC, per-zone — hot-reloadable

Modern Options

  • • Lua is still excellent — fast, tiny, proven in games (WoW, Roblox, Factorio). LuaJIT for performance.
  • • WASM — sandboxed, any source language, near-native speed. Write quests in Rust, Go, or anything that compiles to WASM.
  • • TypeScript (Deno/V8) — if the team knows JS. Better tooling than Lua but heavier runtime.
  • • Data-driven quest graphs — define quests as state machines in data (JSON/YAML), not code. Visual editors possible.
  • • AI-powered NPC dialogue — LLM-generated responses for flavor NPCs. Not for game logic, but for immersion.
ADIF opportunity: Lua is proven and there's no reason to switch from it for quest scripting. The event-driven model (EventNPC, etc.) is excellent — keep that pattern. Layer a data-driven quest graph on top for common quest types (kill X, collect Y, talk to Z) so not every quest needs a script. Consider LLM-powered ambient NPC dialogue as a differentiator.

9. Game Loop & Tick System

KEEP

EQ / EQMacEmu

  • • Fixed-interval timer fires the main loop callback
  • • Sequential entity processing each tick
  • • Custom Timer class with era-based wrap-around handling
  • • BenchTimer for profiling using std::chrono::high_resolution_clock
  • • Zone event scheduler for timed zone-wide events

Modern Refinements

  • • Fixed timestep with interpolation — standard pattern. Server ticks at fixed rate (e.g., 20Hz), client interpolates between states for smooth rendering.
  • • Variable tick rates per system — combat at 10Hz, position at 20Hz, AI decisions at 2Hz. Not everything needs every tick.
  • • ECS scheduling — systems declare dependencies, scheduler runs them in optimal order (potentially parallel)
ADIF opportunity: The fixed-timestep pattern is correct. The main improvement: run different systems at different rates. AI doesn't need 20Hz. Position validation does. This is free performance.

10. Zone Loading & Streaming

RETHINK

EQ / EQMacEmu

  • • Full zone load — client downloads entire zone geometry at once → loading screen
  • • Server boots zone process on demand, loads: .map, .wtr, .nav, then all DB content (spawns, doors, objects, traps, corpses, ground items)
  • • Zone-to-zone: disconnect from zone A, connect to zone B. Full handoff through world server.
  • • "You have entered [Zone Name]" is a real server switch

Modern: Chunk Streaming

  • • Voxel chunks — load/unload 16³ or 32³ chunks around the player. No loading screen for terrain.
  • • Server-side streaming — zone server loads chunks on demand too, not the full world
  • • Seamless zones — if chunks stream, zones can overlap. Players near a border see into the next zone.
  • • Zone boundaries become design choices, not technical limits. Keep them for dungeons (instanced), remove them for overworld.
  • • LOD at distance — lower-resolution voxels for far chunks, full resolution nearby
ADIF opportunity: Voxel chunks eliminate loading screens for the overworld. But keep zone boundaries as a game design tool — dungeons should still be separate instances. The server can manage multiple chunk regions within a zone process, streaming data to clients as they move. This is the biggest UX improvement over classic EQ.

11. Inter-Process Communication

UPGRADE

EQ / EQMacEmu

  • • Memory-mapped files with named mutexes for IPC
  • • MemoryMappedFile class, platform-abstracted (Windows + Unix)
  • • IPCMutex — named binary semaphore for synchronization
  • • All processes must run on the same machine
  • • Tight coupling — shared memory structs must match exactly

Modern Options

  • • gRPC — typed RPC between services, works across machines. Generated from .proto files.
  • • NATS / Redis Pub/Sub — message bus for events (player joined zone, chat message, guild update)
  • • Distributed by default — zone servers can run on different machines. Scale horizontally.
  • • Service mesh — world server becomes a routing layer, not a single point of coordination
ADIF opportunity: gRPC between zone ↔ world for request/response (zone transfers, character loads). Redis Pub/Sub or NATS for broadcast events (chat, guild updates, world announcements). This lets zone servers run on different machines if needed — instant horizontal scaling.

Rethought Server-Side Proof

Since ADIF is its own game, the proof-of-concept is building the first piece of ADIF's own server — not swapping files in EQMacEmu.

What We're Proving

1. Voxel Collision Works

A zone server loads voxel terrain and correctly answers: "where is the ground?", "can I see from A to B?", "is this water?"

2. NPCs Can Navigate

NPCs pathfind through voxel terrain using a NavMesh generated from the voxel surface. They avoid water, walk around obstacles.

3. Zone Transitions Work

A player can zone from voxel zone → mesh zone → back. The world server routes correctly regardless of geometry backend.

For the proof, both zones are voxel. We don't need mesh zones yet — the point is proving the zone server architecture works with voxel geometry, and that zone transitions work between independent zone processes.

Components to Build

Zone Server Process

• Loads voxel terrain from a file (simple format: chunk of voxel IDs)

• Provides collision: ground height lookup, line-of-sight check, material query (solid/water/lava/air)

• Generates NavMesh from voxel surface using Recast/Detour

• Hosts NPCs that patrol waypoints and pathfind when aggro'd

• Accepts client connections, validates position updates, broadcasts entity state

• Reports to world server: "I'm alive, here are my zone points"

World Server Process

• Tracks which zone processes are running and their zone point definitions

• Handles zone transfer requests: "player X wants to go from zone A → zone B"

• Routes cross-zone messages (tells, /who)

• Simple token-based auth (no login server yet — proof-of-concept)

Minimal Client

• Renders voxel terrain (chunk-based, basic lighting)

• First-person or third-person camera

• Sends WASD movement → server validates → client applies authoritative position

• Displays other players and NPCs as simple shapes

• Handles zone transition: disconnect from zone A, connect to zone B, load new terrain

Milestones

1
Voxel terrain + collision

Load a voxel chunk, walk on it. Server validates position (no falling through ground). First-person camera.

2
Multiplayer + NPCs

Second client connects, sees the first player. NPCs spawn and patrol waypoints. Basic aggro radius.

3
Zone transitions

Two zone processes running. Walk to a zone line, get transferred to the other zone. World server coordinates the handoff.

4
The Innothule test

Voxelize the Innothule Swamp layout. Spawn frogloks. Walk through swamp, zone out to a second zone, zone back in. The EQ experience, rebuilt from scratch in voxels.

Decisions Needed Before Building

Server Language

Rust

Performance + safety. bevy_ecs for ECS. Tokio for async. Steep learning curve but prevents entire categories of bugs. Growing gamedev ecosystem.

C# / .NET

Natural fit if using Unity for the client. ASP.NET for the world server. Good async support. Large ecosystem.

C++

Maximum control, same language as EQMacEmu reference. flecs for ECS. But memory safety is on you.

TypeScript / Node

If going browser-based (Three.js/Babylon.js). Same language for client + server. bitECS available. Performance ceiling is lower.

Client / Renderer

Godot + godot-voxel

Open-source, voxel plugin exists (Zylann's godot_voxel). GDScript or C#. Cross-platform. Active community.

Unity + custom voxel

Mature engine, massive asset store. Write custom voxel renderer or use MagicaVoxel-style plugin. C# server + client natural fit.

Three.js / Babylon.js (browser)

No install needed — players connect via URL. WebGL/WebGPU rendering. TypeScript everywhere. Lower performance ceiling but zero friction to play.

Custom engine (Rust + wgpu)

Total control. wgpu for cross-platform GPU access. Most work but most flexibility. Bevy as a framework option.

Voxel Art Style

Blocky (Minecraft-style)

Fast to author, distinctive. Each voxel is a visible cube. Simple but charming. Fastest to get running.

Smoothed (Marching Cubes)

Voxels define a density field, surface extraction creates smooth terrain. More realistic but more complex rendering and authoring.

Hybrid

Smooth terrain + blocky structures. Best of both but harder to make visually consistent.

Networking Protocol

ENet (native UDP)

Proven, lightweight, reliable UDP. Perfect for native clients. C library with bindings for most languages.

WebSocket + WebRTC

WebSocket for reliable (chat, inventory), WebRTC data channels for unreliable (position). Required for browser clients.

QUIC

Modern UDP with built-in encryption and multiplexed streams. More complex but future-proof.
